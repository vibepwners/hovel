# Mbed OS NaCl proof of concept

This application runs the real Cortex-M4 hosted `nacl_server` or `nacl_client`
PIC blob on a NUCLEO-F429ZI under Mbed OS 5.15.9. The platform adapter maps the
blob vtable to Mbed TCP sockets, retargeted console I/O, and the target TRNG.

The data channel uses TweetNaCl `crypto_secretbox` (XSalsa20-Poly1305) with the
same per-session symmetric key in both directions. The client sends encrypted
application data and the server returns an encrypted `OK`; each side
authenticates and decrypts its inbound frame before reporting success.

## Generate configured blob headers

Generate one random 32-byte authentication key and keep the same key for the
server and client firmware. Materialize both headers through Task, substituting
the server board's IPv4 address:

```text
task picblobs:mbed-blobs -- \
  --server-ip=192.0.2.10 \
  --auth-key-file=/secure/path/mbed-poc.key \
  --port=9999
```

The command writes ignored files under `mbed/blobs/`. It rejects an all-zero or
wrong-sized key. The file may contain 32 raw bytes or 64 hexadecimal
characters. `--auth-key-hex` is also available for disposable lab keys, but it
exposes the key in shell and process arguments. The key is deployment
configuration and is not part of the checked-in blob source; the generated
firmware will necessarily contain it. Use a distinct random key for every
assessment or lab deployment.

## Build and run

The checked-in Mbed project pins Mbed OS 5.15.9 and targets NUCLEO-F429ZI with
GCC_ARM. Build one firmware with `blob_role` set to `"server"`, then build the
other with it set to `"client"`. Both builds must use headers generated by the
same materialization command. Connect both boards to the same authorized lab
network and start the server before the client.

Deploy the exact SDK named by `mbed-os.lib`, then run the repository's
Task-backed target build:

```text
task picblobs:mbed-compile
```

The task uses the pinned GCC ARM 9-2019-q4 toolchain supported by Mbed OS 5.15,
validates the checked-out SDK version, and produces separate server and client
firmware under `.local/picblobs/mbed/NUCLEO_F429ZI/GCC_ARM/`. Use
`--role=server` or `--role=client` to build only one role.

The lightweight headers under `mbed/picblobs/` forward to the canonical hosted
platform ABI in `src/include/picblobs/`, keeping the Mbed project buildable
without copying that ABI.

Expected serial output includes:

```text
[server] decrypted: Hello from NaCl PIC blob!
[server] secure channel OK
```

and:

```text
[client] decrypted ACK: OK
[client] secure channel OK
```

Repository CI cross-compiles the actual Cortex-M4 blobs and executes them
through `platform_mbed.cpp` with POSIX-backed Mbed API mocks. This validates the
blob machine code, socket adapter, TRNG contract, and bidirectional encrypted
exchange without claiming to replace a physical-board smoke test.
