Squatter Payload Provider

Squatter is Hovel's first core payload provider. It is a lab and demonstration payload system for authorized environments. The first target is Windows XP SP3 x86. The provider is a normal Hovel payload_provider module that speaks JSON-RPC over stdio; the Windows payload binary is opaque payload code produced by that provider.

Hovel must not learn Squatter's implant wire protocol, command grammar, PE internals, or Windows API strategy. Hovel learns step contracts, payload identity, compatibility, lifecycle requirements, artifacts, transport endpoints, payload instances, cleanup handles, and session refs from the provider over RPC. Hovel controls planning, confirmation, artifact recording, capability validation, and session brokering; the provider owns Squatter's transport and wire semantics.

Settled Decisions

  1. Squatter is a payload capability, not a native Hovel execution path.
  2. The Squatter provider is a first-class payload_provider module with explicit chain steps.
  3. The provider reports step contracts, configuration schema, prepare materialization, cleanup contracts, and output capabilities through JSON-RPC.
  4. Hovel discovers payload formats, transports, compatibility, and session strategy from live provider contracts, not static payload YAML.
  5. Final payload bytes are generated during execution after confirmation, then recorded as artifacts with hashes.
  6. Hovel materializes provider bytes to controlled workspace artifact paths and passes typed payload artifacts to installer steps.
  7. Payload generation and materialization are per target and per transport.
  8. A provider may return multiple artifacts for one target, but one artifact must be marked primary.
  9. Exploit modules do not receive Squatter protocol details. They produce execution capabilities that Squatter install steps consume.
  10. Provider compatibility is checked during planning from contracts and metadata; actual bytes are generated after confirmation.
  11. The provider is tagged dangerous because Squatter supports file transfer, process execution, DLL loading, task listing, and sessions.
  12. Initial communication is unencrypted. Encryption is provider-reported metadata and appears in review/audit output, but it is not a separate operator config field.

The provider is written in Go using Hovel's Go SDK. The Windows payload is C code built with a pinned MinGW cross toolchain. The first implementation may produce a nonfunctional minimal PE placeholder, but it must exercise the real Windows build path.

Module Contract

A payload provider participates in the common module lifecycle and exposes chain-step contracts through the generic step RPC methods: step.describe, step.prepare, step.execute, and step.cleanup. The provider dispatches internally by step_id. Hovel never calls Squatter-specific private methods for orchestration. 

squatter.generate
squatter.install_smb
squatter.connect_smb
squatter.install_tcp_bind
squatter.connect_tcp_bind
squatter.listen_tcp_callback
squatter.install_tcp_callback
squatter.connect_tcp_callback
squatter.cleanup_smb
squatter.cleanup_tcp_bind
squatter.cleanup_tcp_callback

Only connector and listener steps produce SessionRef. Generate and install steps may produce PayloadArtifact, PayloadInstance, TransportEndpoint, and CleanupHandle, but they do not open operator sessions.

Provider Configuration

Common options should use payload.* keys. Squatter-specific options may use squatter.* keys when they are not portable across payload providers. 

payload.provider = squatter
payload.format = pe-exe
payload.transport = smb-named-pipe | tcp-bind | tcp-callback
payload.lhost = 10.10.14.3
payload.lport = 4444
payload.bind_port = 9100
payload.pipe = random-neutral
squatter.agent.max_chunk_size = 65536

payload.transport is required. The first Squatter provider supports smb-named-pipe, tcp-bind, and tcp-callback. SMB named-pipe mode requires an SMB CredentialCapability. TCP bind mode requires a target endpoint. TCP callback mode requires a listener step and may remain in installed_waiting_callback until the agent connects back.

Payload Metadata

Payload metadata is returned by the provider over RPC. Hovel uses it for planning, review, compatibility checks, artifact recording, and session orchestration. 

{
  "id": "squatter/windows/x86/xp-sp3/smb-named-pipe/pe-exe",
  "name": "squatter",
  "version": "0.1.0",
  "platform": "windows",
  "arch": "x86",
  "minOS": "windows-xp-sp3",
  "testedOS": ["windows-xp-sp3"],
  "formats": ["pe-exe"],
  "capabilities": [
    "file.get",
    "file.put",
    "process.exec",
    "process.tasklist",
    "library.rundll"
  ],
  "transport": {
    "kind": "smb-named-pipe",
    "encrypted": false
  },
  "session": {
    "kind": "agent",
    "acquisition": "active-connect",
    "requiresPreThrowListener": false,
    "requiresPostThrowConnect": true,
    "owner": "payload_provider"

}

DLL output is part of the contract, but the first build target should focus on a PE EXE. DLL/rundll support adds export and loader semantics and should follow once provider orchestration is stable.

Transports

SMB named-pipe, TCP bind, and TCP callback comms are all required Squatter functionality. They differ in chain requirements and reconnect strategy, but the Squatter provider owns the actual LP-to-payload protocol in all modes.

SMB named-pipe mode uses credential-backed active connection:

squatter.generate
squatter.install_smb
squatter.connect_smb
provider-owned Hovel session

TCP bind mode uses target-side listening:

squatter.generate
squatter.install_tcp_bind
squatter.connect_tcp_bind
provider-owned Hovel session

TCP callback mode uses a confirmed listener step:

squatter.listen_tcp_callback
squatter.generate
squatter.install_tcp_callback
payload callback
squatter.connect_tcp_callback
provider-owned Hovel session

SMB mode uses a Windows named pipe exposed through SMB. The provider connects to \\target\pipe\name using an explicit CredentialCapability and exposes the resulting Squatter agent session through Hovel's session abstraction. The exploit module does not own or proxy that session. The operator-side connector uses github.com/jfjallid/go-smb to authenticate to IPC$, open the configured pipe, and run the same Squatter mux protocol used by TCP transports over the resulting pipe handle. The required configuration keys are target.host or smb.host, payload.pipe, smb.username, smb.password, optional smb.domain, and optional smb.port.

The first SMB connector is SMB2/3 only because it is backed by github.com/jfjallid/go-smb. Windows XP targets that only expose SMBv1 should use tcp-bind or tcp-callback for the representative Etro chain unless a separate SMBv1 pipe connector is added.

TCP bind mode patches the payload to listen on payload.bind_port on the target. The provider connects to target.host:payload.bind_port after installation and emits an active SessionRef when the TCP connection opens. TCP callback mode patches the payload with payload.lhost and payload.lport, starts a provider listener, and adopts the accepted connection as the Squatter session.

Prepared Target Footprint

Prepare is pure/local. It generates planned target-side identifiers and stores them before confirmation, but it does not build the final executable, contact the target, stage files, or start services. The operator may edit prepared values before confirmation; after confirmation they are immutable. 

credential username: random neutral string
credential password: random high-entropy string
service name:       random neutral string
service display:    random neutral or configured string
staged filename:    random neutral .exe by default
pipe name:          random neutral string
collision strategy: fail after confirmation

Target-side names must not include Hovel, Squatter, run ids, or other tool-identifying strings. Evidence stores the complete mapping plainly, including plaintext created credentials, so the operator can remediate even if automated cleanup fails.

Session Ownership

Squatter sessions are provider-managed and Hovel-brokered. A live install is a durable PayloadInstance; a session is an attachable view over that instance. Hovel sees a SessionRef with kind, transport, state, payload instance id, and capabilities. The provider owns framing, task IDs, file chunking, command/result mapping, retries, and transport details. 

session kind        agent
session owner       payload_provider
transport           squatter/smb-named-pipe, squatter/tcp-bind, or squatter/tcp-callback
capabilities        file.get, file.put, process.exec, process.tasklist, library.rundll

The provider may present a shell-like operator experience through Hovel's session API, but the LP-to-payload protocol should be typed and robust. Hovel should not treat a Squatter session as a raw cmd.exe terminal.

Operators may detach from a live Squatter session and reconnect later when the transport supports it. SMB named-pipe and TCP bind reconnect with an active-connect strategy. TCP callback sessions attach to an accepted listener connection; if no callback is present, the instance remains installed_waiting_callback or disconnected according to provider observation.

Throw Planning

The throw plan hash covers operator intent, the live step contract snapshot, and deterministic configuration: selected provider steps, selected transport, provider config, prepared target-side names, expected format, expected platform, compatibility facts, and whether listener or connector steps are required.

The plan hash should not include final generated payload bytes, runtime session IDs, or observed connection handles. Those values are produced after confirmation and recorded as events and artifacts. If the provider reports a different contract after confirmation, execution is blocked until the operator reviews the diff, re-prepares if needed, and re-confirms.

Chain validation must enforce declared requirements. Validation should fail when a Squatter install step lacks a compatible payload artifact, when SMB mode lacks an SMB credential capability, or when a connector step lacks an installed payload instance and endpoint in an allowed state.

Throw Execution

Squatter steps produce capabilities, artifacts, evidence, and state transitions. Generation failures, listener bind failures, install failures, and connection failures are reported at the responsible step with exact evidence. 

for each target:
  query live step contracts
  prepare local names and transport config
  confirm snapshot
  generate final per-target payload artifacts
  materialize artifacts in workspace
  install payload with declared capabilities
  connect or listen for session when required
  keep provider alive while it owns active sessions
  cleanup target and provider-side state from cleanup handles

Hovel records generated payloads in the normal artifact list. Operators should see metadata such as name, kind, provider module, target, throw, SHA-256, size, and controlled workspace path. Hovel should not render binary bytes in ordinary terminal output.

If install succeeds but session connection fails, Squatter returns a PayloadInstance in installed_unconnected state and a warning evidence record. The CLI must print the exact target, staged path, service name, pipe or port, plaintext SMB credential when one was created, connection endpoint, exact error, active cleanup handles, and the reconnect and cleanup commands.

Build Contract

Squatter is part of Hovel's default build graph. task build should build the provider and the Windows payload target, and task ci should eventually validate both through the root build and test suites.

The Windows payload uses a repo-local Bazel rule or macro over a pinned MinGW archive: 

url = "https://github.com/Vibe-Pwners/windows-toolchains/releases/download/v0.0.1/mingw-w64-gcc-15.2.0-ucrt-posix-dw2-i686-w64-mingw32-linux-x86_64.tar.xz"
sha256 = "<recorded checksum>"

The first build helper should be narrow, for example mingw_pe_binary(...), instead of introducing a broad C/C++ toolchain integration before multiple payloads need it. 

mingw_pe_binary(
    name = "squatter_x86_exe",
    srcs = ["src/main.c"],
    out = "squatter.exe",
    entry = "squatter_entry",
)

Defaults should encode the intended no-CRT/no-import posture: 

-ffreestanding
-fno-stack-protector
-fno-asynchronous-unwind-tables
-nostdlib
-nodefaultlibs
-Wl,--entry,<entry>

A PE inspection test should verify the produced placeholder is i386, has the expected subsystem and entry point, and has no unexpected import table. That test protects the XP SP3 portability goal before the real PEB-based resolver exists.

Windows Payload Contract

The Windows payload is not cross-platform code. The first target is XP SP3 x86. The implementation should eventually resolve Win32 APIs through the PEB rather than relying on a normal import table or CRT startup. The provider should report conservative compatibility until broader Windows versions are tested.

Base Squatter payload capabilities:

  1. Get a file from the target.
  2. Put a file onto the target.
  3. Drop and run an EXE.
  4. Drop and run a DLL through rundll-style semantics.
  5. Return a task list.

IOCP/threading, the event loop, the agent command implementation, the LP network protocol, and the PEB API resolver are intentionally deferred until the provider interface and build system are documented and scaffolded.

Sprint Implementation Plan

This sprint turns Squatter from a payload byte generator into a representative chain participant with install, connect, operate, reconnect, and cleanup semantics.

The implementation pass should include:

  1. Implemented: core capability schema names for RemoteExecutionCapability, CredentialCapability, PayloadArtifact, PayloadInstance, TransportEndpoint, SessionRef, and CleanupHandle.
  2. Implemented: generic step JSON-RPC dispatch: step.describe, step.prepare, step.execute, and step.cleanup.
  3. Partially implemented: live contract planning now resolves enabled module steps to capability bindings or typed missing requirements; contract snapshot persistence and post-confirmation drift blocking remain pending.
  4. Partially implemented: Squatter SMB install prepare materializes random neutral service names, staged filenames, pipe names, endpoint config, and planned cleanup handles. Credential creation remains a separate module step.
  5. Partially implemented: SMB named-pipe connect uses go-smb with explicit credentials and emits a SessionRef capability. Full Hovel interactive session brokering over that pipe remains pending.
  6. Partially implemented: TCP bind payload generation patches an explicit bind port, the provider can actively connect to that endpoint, and squatter.connect_tcp_bind emits a SessionRef capability.
  7. Partially implemented: TCP callback listener and connector steps start a provider listener, adopt an accepted payload connection, and emit a SessionRef capability.
  8. Final Squatter artifact generation during execute, with hash and artifact evidence.
  9. A module for connecting to an already-installed Squatter over SMB with operator-supplied credentials.
  10. Durable PayloadInstance state and detach/reconnect behavior.
  11. Warning-state reporting for installed_unconnected with exact CLI evidence.
  12. Cleanup handles for created credentials, staged files, services, listeners, and payload instances, with retry and verification where possible.

The implementation pass should not include:

  1. Etro-specific Squatter protocol handling.
  2. Hidden helper-step insertion by the planner.
  3. Target-visible Hovel or Squatter ownership markers.
  4. Command-only Squatter staging as a fallback for the representative SMB chain.
  5. Silent credential redaction in normal operator CLI output.

The representative Etro chain should require an upload-capable remote execution path for SMB install. If Etro cannot honestly produce an upload_file sub-capability, Squatter SMB install must not pretend it can use Etro for staging. A separate command-only installer variant may be designed later if it becomes operationally necessary.
