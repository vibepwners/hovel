Squatter Payload Provider

Squatter is Hovel's first core payload provider. It is a lab and demonstration payload system for authorized environments. The first target is Windows XP SP3 x86. The provider is a normal Hovel payload_provider module that speaks JSON-RPC over stdio; the Windows payload binary is opaque payload code produced by that provider.

Hovel must not learn Squatter's implant wire protocol, command grammar, PE internals, or Windows API strategy. Hovel learns payload identity, compatibility, lifecycle requirements, artifacts, and session refs from the provider over RPC. Hovel controls where received payload bytes are written, records hashes and metadata, and passes paths to exploit modules.

Settled Decisions

  1. Squatter is a payload capability, not a native Hovel execution path.
  2. The Squatter provider is a first-class payload_provider module and appears as an explicit chain step.
  3. The payload provider reports configuration through schema(), so options appear in normal interactive chain configuration.
  4. Hovel discovers payloads, formats, capabilities, and session strategy from provider RPC calls, not static payload YAML.
  5. Payload bytes are delivered from provider to Hovel over JSON-RPC. Initial responses may carry base64 bytes; the contract should allow chunked reads for larger payloads.
  6. Hovel materializes provider bytes to controlled workspace artifact paths and passes local paths plus metadata to exploit modules.
  7. Payload generation and materialization are always per target.
  8. A provider may return multiple artifacts for one target, but one artifact must be marked primary.
  9. Exploit modules receive a typed payloads field in execute requests rather than user-supplied chain config keys.
  10. Provider compatibility is checked during planning from metadata only; actual bytes are generated after confirmation.
  11. The provider is tagged dangerous because Squatter supports file transfer, process execution, DLL loading, task listing, and sessions.
  12. Initial communication is unencrypted. Encryption is provider-reported metadata and appears in review/audit output, but it is not a separate operator config field.

The provider is written in Go using Hovel's Go SDK. The Windows payload is C code built with a pinned MinGW cross toolchain. The first implementation may produce a nonfunctional minimal PE placeholder, but it must exercise the real Windows build path.

Module Contract

A payload provider still participates in the common module lifecycle: handshake, schema, provider-specific calls, and shutdown. It must also expose provider methods through the same Content-Length framed JSON-RPC channel. 

list_payloads(query) -> []PayloadInfo
resolve_payload(query) -> PayloadInfo
prepare_listener(request) -> ListenerRef | null
generate_payload(request) -> PayloadArtifactSet
connect_session(request) -> SessionRef | null
cleanup_payload(request) -> CleanupResult
read_payload_chunk(request) -> PayloadChunk

execute remains the normal module-run method, but Hovel should use provider-specific methods for payload lifecycle orchestration. This keeps payload generation distinct from survey or exploit execution.

Provider Configuration

Common options should use payload.* keys. Squatter-specific options may use squatter.* keys when they are not portable across payload providers. 

payload.provider = squatter
payload.format = pe-exe
payload.transport = reverse-tcp | smb-named-pipe
payload.lhost = 10.10.14.3
payload.lport = 4444
payload.pipe = hovel-squatter-<target>
squatter.agent.max_chunk_size = 65536

payload.transport is required. The first Squatter provider supports reverse-tcp and smb-named-pipe. Reverse TCP requires listener host and port configuration. SMB named-pipe mode requires pipe configuration and connects to the target after the exploit runs.

Payload Metadata

Payload metadata is returned by the provider over RPC. Hovel uses it for planning, review, compatibility checks, artifact recording, and session orchestration. 

{
  "id": "squatter/windows/x86/xp-sp3/reverse-tcp/pe-exe",
  "name": "squatter",
  "version": "0.1.0",
  "platform": "windows",
  "arch": "x86",
  "minOS": "windows-xp-sp3",
  "testedOS": ["windows-xp-sp3"],
  "formats": ["pe-exe"],
  "capabilities": [
    "file.get",
    "file.put",
    "process.exec",
    "process.tasklist",
    "library.rundll"
  ],
  "transport": {
    "kind": "reverse-tcp",
    "encrypted": false
  },
  "session": {
    "kind": "agent",
    "acquisition": "callback",
    "requiresPreThrowListener": true,
    "requiresPostThrowConnect": false,
    "owner": "payload_provider"
  }
}

DLL output is part of the contract, but the first build target should focus on a PE EXE. DLL/rundll support adds export and loader semantics and should follow once provider orchestration is stable.

Transports

Reverse TCP and SMB named-pipe comms are both required Squatter functionality. They differ only in provider lifecycle from Hovel's point of view; the Squatter provider owns the actual LP-to-payload protocol in both cases.

Reverse TCP uses pre-throw listener acquisition:

resolve_payload
prepare_listener
generate_payload
exploit execute with payload path
payload callback
provider-owned Hovel session

SMB named-pipe mode uses post-throw connection acquisition:

resolve_payload
generate_payload
exploit execute with payload path
connect_session
provider-owned Hovel session

SMB mode should use a Windows named pipe for comms through SMB. Hovel does not implement the SMB transport. The provider connects to the named pipe and exposes the resulting session through Hovel's session abstraction.

Session Ownership

Squatter sessions are provider-managed and Hovel-brokered. Hovel sees a SessionRef with kind, transport, state, and capabilities. The provider owns framing, task IDs, file chunking, command/result mapping, retries, and transport details. 

session kind        agent
session owner       payload_provider
transport           squatter/reverse-tcp or squatter/smb-named-pipe
capabilities        file.get, file.put, process.exec, process.tasklist, library.rundll

The provider may present a shell-like operator experience through Hovel's session API, but the LP-to-payload protocol should be typed and robust. Hovel should not treat a Squatter session as a raw cmd.exe terminal.

Throw Planning

The throw plan hash covers operator intent and deterministic configuration: selected provider step, selected payload query, selected transport, provider config, target config, expected format, expected platform, compatibility facts, and whether pre/post actions are required.

The plan hash should not include generated payload bytes, runtime session IDs, listener handles, or dynamically assigned connection details unless the operator explicitly pinned them in config. Those values are produced after confirmation and recorded as events and artifacts.

Chain validation must enforce provider-before-exploit ordering for exploits that consume payloads. Validation should fail when an exploit declares a payload requirement and no compatible payload provider step appears earlier in the chain.

Throw Execution

A provider step produces a normal Hovel run result plus artifacts and events. Payload preparation failures, listener bind failures, generation failures, and post-throw connection failures should be visible as provider-step failures. 

for each target:
  run payload provider step
  resolve payload metadata
  prepare listener if required
  generate per-target payload artifacts
  materialize artifacts in workspace
  run exploit with typed payload path metadata
  connect session if required
  keep provider alive while it owns active sessions
  cleanup provider-side state after session/throw cleanup

Hovel records generated payloads in the normal artifact list. Operators should see metadata such as name, kind, provider module, target, throw, SHA-256, size, and controlled workspace path. Hovel should not render binary bytes in ordinary terminal output.

Build Contract

Squatter is part of Hovel's default build graph. task build should build the provider and the Windows payload target, and task ci should eventually validate both through the root build and test suites.

The Windows payload uses a repo-local Bazel rule or macro over a pinned MinGW archive: 

url = "https://github.com/Vibe-Pwners/windows-toolchains/releases/download/v0.0.1/mingw-w64-gcc-15.2.0-ucrt-posix-dw2-i686-w64-mingw32-linux-x86_64.tar.xz"
sha256 = "<recorded checksum>"

The first build helper should be narrow, for example mingw_pe_binary(...), instead of introducing a broad C/C++ toolchain integration before multiple payloads need it. 

mingw_pe_binary(
    name = "squatter_x86_exe",
    srcs = ["src/main.c"],
    out = "squatter.exe",
    entry = "squatter_entry",
)

Defaults should encode the intended no-CRT/no-import posture: 

-ffreestanding
-fno-stack-protector
-fno-asynchronous-unwind-tables
-nostdlib
-nodefaultlibs
-Wl,--entry,<entry>

A PE inspection test should verify the produced placeholder is i386, has the expected subsystem and entry point, and has no unexpected import table. That test protects the XP SP3 portability goal before the real PEB-based resolver exists.

Windows Payload Contract

The Windows payload is not cross-platform code. The first target is XP SP3 x86. The implementation should eventually resolve Win32 APIs through the PEB rather than relying on a normal import table or CRT startup. The provider should report conservative compatibility until broader Windows versions are tested.

Base Squatter payload capabilities:

  1. Get a file from the target.
  2. Put a file onto the target.
  3. Drop and run an EXE.
  4. Drop and run a DLL through rundll-style semantics.
  5. Return a task list.

IOCP/threading, the event loop, the agent command implementation, the LP network protocol, and the PEB API resolver are intentionally deferred until the provider interface and build system are documented and scaffolded.

First Implementation Slice

The first code pass should include:

  1. Go SDK payload-provider types and JSON-RPC method dispatch.
  2. A stub Squatter Go provider module.
  3. Provider RPC contract tests.
  4. A pinned repo-local MinGW Bazel helper.
  5. A minimal Windows XP x86 PE placeholder target.
  6. A PE inspection test for machine type, subsystem, entry point, and imports.
  7. Root build wiring so Squatter participates in task build.

The first code pass should not include:

  1. Full throw orchestration for provider lifecycle hooks.
  2. Real Squatter IOCP/threading or command handlers.
  3. Real reverse TCP listener implementation.
  4. Real SMB named-pipe transport implementation.
  5. PEB API lookup implementation.

The second pass wires provider-before-exploit validation, plan/review compatibility, per-target generation and materialization, typed execute payloads, pre-listener and post-connect lifecycle hooks, and provider-owned sessions.
