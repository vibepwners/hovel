Testing and Roadmap

Testing should focus on contracts first. The risk is not only individual functions breaking; it is adapters disagreeing about the same operation, chain, throw, service, provider, or event.

Go Host Tests

Require:

  1. Unit tests for domain logic.
  2. Application service tests.
  3. Adapter tests with fake services.
  4. Module protocol contract tests.
  5. Service protocol contract tests once the general service process runtime exists.
  6. Supervisor tests for daemon and module process ownership, plus service supervision when that runtime lands.
  7. Golden tests for one-shot chain-file output.
  8. CLI prompt model, completion, and render tests.
  9. TUI model/update tests.

Python SDK Tests

Require:

  1. Logging handler tests.
  2. RPC protocol tests.
  3. HovelModule metadata, schema, and class-attribute tests.
  4. Step hook dispatch tests for step.describe, step.prepare, step.execute, and step.cleanup.
  5. Session and installed-payload descriptor serialization tests.
  6. Packaging tests.

Integration Tests

First-slice integration tests:

  1. Host launches Python module.
  2. Module handshakes.
  3. Module emits logs.
  4. Module returns result.
  5. Host treats module stdout as the framed RPC transport, records SDK module/log notifications as structured events, and preserves stderr as failure diagnostics.
  6. Host records malformed protocol output as a failed module execution.
  7. One-shot chain-file execution displays throw results.
  8. hovel starts or attaches to hoveld.
  9. One-shot chain-file execution crosses the daemon boundary.
  10. Throw plan, confirmation record, events, artifacts, and final throw state are persisted.
  11. Explicit installed-payload descriptors persist current-state rows and append-only state events.
  12. payloads installed reads SQLite inventory without probing targets.
  13. payloads connect <id> uses the provider reconnect descriptor and creates a normal Hovel session on success.
  14. Interactive throw, review, confirm, and one-shot chain-file execution share the same planning path, while review and prompted throw share the same typed review path.
  15. throw --now persists an auditable bypass confirmation record.
  16. If cli or tui starts a managed daemon, it stops that daemon on exit.
  17. If cli or tui attaches to an existing daemon, it leaves that daemon running on exit.
  18. A cli session can select an operation and chain, add operation targets, and issue throw using attachment state.
  19. chain save <file> and chain load <file> preserve both targetless chain templates and configured chains with target config.
  20. Two daemon clients attached to the same operation share operation target inventory while keeping active chains, logs, and prompt context separate.
  21. One-shot execution of a saved chain file does not mutate another client's active chain.
  22. The shared throw-control application service records a persisted plan, requires exact plan-hash confirmation, and gates throw start on confirmation and launch-key readiness.
  23. Go, Python, and Rust SDKs expose optional agent context only when supplied and omit agentHints unless a module opts in.

Follow-on integration tests:

  1. Module requests a provider resource.
  2. Host launches managed service.
  3. Service handshakes.
  4. Service health check passes.
  5. Service provides payload or listener resource.
  6. Provider version or reconnect-schema incompatibility keeps installed payload records visible while marking reconnect unavailable.
  7. The stdio MCP adapter attaches to hoveld as an MCP operator entity, heartbeats, exposes typed read-only inspection tools, and detaches on shutdown.
  8. MCP can inspect and plan the same throw as one-shot execution and cli through shared guardrails.
  9. MCP can execute only through the shared confirmation path.
  10. CLI and one-shot throw execution use the shared throw-control service instead of parallel planning or confirmation logic.
  11. Launch-key mode blocks a pending throw until every required live operation entity approves the same plan hash.
  12. End-to-end daemon execution persists agent hints with provenance and presents them through MCP resources or prompt material without granting authority.
  13. TUI consumes the structured logging rail.

Coverage Targets

Initial targets:

Domain: 90%+
Application services: 85%+
Adapters: practical coverage, not vanity coverage
Python SDK: 85%+

Strict branch coverage gates for core packages.

Milestones

The milestone list is a planning roadmap, not a live feature matrix. Some later slices are already present in source, including a first MCP inspection adapter. Service lifecycle management, the TUI, REST, MCP execution parity, and richer chain-control features remain future work.

Milestone 1: Skeleton

  1. Create Go repo skeleton.
  2. Add hovel mono-binary root.
  3. Add hovel daemon engine skeleton.
  4. Add domain packages.
  5. Add event model.
  6. Add module descriptor schema.
  7. Add service descriptor schema.
  8. Add local workspace initialization under control init.
  9. Add in-process application service test harness.
  10. Add daemon status under control daemon status and workspace locking contract.

Milestone 2: Module Runtime

  1. Implement JSON-RPC-over-stdio module host with Content-Length framing.
  2. Implement Python SDK handshake.
  3. Implement Python logging handler.
  4. Run a toy Python module from Go.
  5. Stream logs into one-shot execution and cli.
  6. Persist module execution events.
  7. Treat malformed frames and unexpected stdout bytes as module failures.

Milestone 3: Throw and Artifact Persistence

  1. Implement persisted throw plans.
  2. Implement minimal confirmation records keyed by plan hash.
  3. Implement throw state persistence.
  4. Implement artifact provider.
  5. Hash-track artifacts and payload-like bytes, storing large artifact bytes outside SQLite.
  6. Inspect throws and artifacts from cli and supported API surfaces.

Milestone 4: Providers

  1. Implement payload provider interface.
  2. Add local payload registry.
  3. Resolve tagged bytes from local or stub providers.
  4. Persist payload metadata and hashes before execution when practical.
  5. Return payload refs to Python modules.
  6. Implement stamped payload artifact records and explicit installed-payload descriptor persistence.
  7. Add payloads available, payloads installed, payloads inspect, payloads connect, payloads cleanup, and payloads mark-removed.
  8. Add the Squatter stub provider, provider RPC contract tests, pinned Windows PE build target, and PE inspection test before wiring full throw orchestration.

Milestone 5: Service Manager

  1. Implement supervised service process manager.
  2. Implement service descriptor loading.
  3. Implement service handshake.
  4. Implement service health checks.
  5. Implement toy service that emits logs.

Milestone 6: Chain Thrower

  1. Implement simple sequential chain runner.
  2. Add phases and steps.
  3. Add chain template and configured chain save/load.
  4. Add service start/stop steps.
  5. Add structured logging rail.
  6. Add per-target throw state.
  7. Add cancellation hooks.

Milestone 7: Conceptual Chain Demo

Milestone 8: TUI Alpha

Hard Rules

  1. cli and one-shot chain-file execution first, TUI second.
  2. hoveld is mandatory and owns operations, chains, throws, module processes, structured events, module-backed providers, installed payload inventory, artifacts, and sessions. General service lifecycle ownership starts when the service runtime lands.
  3. TUI calls application services only.
  4. MCP calls application services only.
  5. REST calls application services only.
  6. MCP has eventual operator parity through the same guardrail and audit model.
  7. Throw execution requires a persisted plan and confirmation record.
  8. Python module authors should not need to see transport internals.
  9. Future Python service authors should not need to see transport internals.
  10. All module logs flow through the host.
  11. All future service logs flow through the host.
  12. Providers are typed.
  13. First-slice providers are artifact and payload only.
  14. Payload provider outputs are tagged bytes with minimal framework validation.
  15. Installed payload records are created only from explicit descriptors and persisted by Hovel core.
  16. Services are lifecycle-managed once the service runtime lands.
  17. Payloads are content-addressed or hash-tracked.
  18. Chain throws are replay-inspectable.
  19. Public demos are authorized and lab-safe.
