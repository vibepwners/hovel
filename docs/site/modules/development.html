Build a Module or Provider

This is the Modules-side starting point for implementers. The module development book explains the shared process and wire protocol. The Mesh and credential guide contains the complete provider contracts and working Go, Python, and Rust examples.

Pick the smallest contract

  1. Implement ordinary module metadata and handshake.
  2. Add only the Mesh surfaces your provider supports: stream, topology, beacons, tasks, triggers, or listeners.
  3. Advertise credentialDelivery only when the provider consumes Hovel-managed credentials.
  4. Use separate listening-post and provider contracts when they have different lifecycles.
  5. Keep provider protocol details behind the SDK contract; consumer modules should use ordinary session or local bridge interfaces.

Provider completion checklist

  • The descriptor reports only implemented capabilities and uses explicit size bounds.
  • Unknown fields, unsupported variants, malformed IDs, and oversized inputs fail closed.
  • Runtime credential hooks retain material only for the consuming operation and never echo it.
  • Mesh tasking preserves Hovel's plan, confirmation, dangerous-module, and launch-key safeguards at the front end.
  • Local bridge consumers authenticate with the returned bearer capability and support the declared TCP or UDP semantics.
  • Tests run through task and pass the repository's hermetic gate.
  • The module gets an operator-facing Modules space with overview, use, build, test, and reference pages.

What Hovel owns

Hovel owns workspace PKI custody, assignment lifecycle, provider discovery, daemon authorization, operation bookkeeping, secret-free execution evidence, and local bridge ownership. Your provider owns its remote protocol, topology, routes, native task execution, listener runtime, and provider-side state. The SDK is the boundary between them; it does not implement a tunnel or force every provider to support every capability.
